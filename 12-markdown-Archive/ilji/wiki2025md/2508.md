## PCB 기판 청소는
먼지 제거, 오염 물질 제거, 부식 제거 단계로 나눌 수 있으며, 주로 압축 공기, 이소프로필 알코올(IPA), 정전기 방지 브러시를 사용합니다. 먼저 압축 공기로 큰 먼지를 제거하고, 이소프로필 알코올과 브러시로 찌든 때나 플럭스 잔여물을 닦아낸 후, 보푸라기 없는 천으로 물기를 닦고 완전히 건조시키는 것이 중요합니다. 
준비물 압축 공기 캔, 이소프로필 알코올(IPA), 정전기 방지 브러시 또는 면봉, 보푸라기 없는 천, 부식이나 오염이 심한 경우 지우개. 
### 청소 방법
#### 1. 먼지 제거:
- PCB 보드의 전원을 끄고, 전원 플러그를 뽑은 상태에서 시작합니다. 
- 압축 공기 캔을 PCB에서 4~6인치 떨어진 곳에 대고, 공기 흐름으로 먼지나 이물질을 날려 보냅니다. 
#### 2. 찌든 때 및 잔여물 제거:
- 정전기 방지 브러시나 면봉에 이소프로필 알코올을 적셔 PCB 표면에 묻은 찌든 때나 플럭스 잔여물을 부드럽게 문질러 닦아냅니다. 
- 부식된 부분이 있다면, 붓이나 지우개를 사용하여 부드럽게 닦아냅니다. 
#### 3. 건조:
- 알코올이 완전히 마르도록 PCB 보드를 충분히 건조시킵니다. 
- 보푸라기 없는 천으로 과도한 액체를 닦아내고 자연 건조하는 것이 좋습니다. 
#### 주의사항
PCB 보드는 민감하므로 너무 강한 압력으로 문지르거나 긁지 않도록 주의합니다. 
물로 세척하는 경우, 물기가 완전히 마르지 않으면 쇼트가 발생하여 부품이 손상될 수 있으므로 충분히 건조해야 합니다. 
보드 전체를 담그는 방식은 부품 종류에 따라 다를 수 있으므로, 가능하면 특정 부분만 청소하는 것이 안전합니다.


```bash
npx create-svelte my-svelte-app
```

## 🔥 bash 에서 원격서버에 접속해서 두가지의 명령을 수행하려고 해.
내가 원하는건, ssh -p host_port host_user@host_dir/ 명령으로 원격 서버에 접속해서 ls -lhR local_dir/ 명령과 rclone lsl cloud_host:backup_dir/ 명령으로 rclone 작업을 실행하려고 하는데, 이걸 bash 셀에서 한줄로 실행하려면 어떻게 하지?

🔋
### 🔋 하나의 ssh 명령으로 여러 명령을 실행하려면,
원격 서버에서 실행할 명령들을 세미콜론(;)으로 구분하거나 논리 연산자(&& 또는 ||)를 사용하여 연결하면 됩니다.

### 명령어 구성
원격 서버에서 두 개의 명령어를 순차적으로 실행하려면, ssh 명령 뒤에 따옴표(" ")로 묶어 두 명령어를 ;로 연결하면 됩니다.
```Bash
ssh -p host_port host_user@host_dir/ "ls -lhR local_dir/ ; rclone lsl cloud_host:backup_dir/"
```
- `ssh -p host_port host_user@host_dir/`: 원격 서버에 접속하는 기본 명령입니다.
- `"..."`: 이 따옴표 안에 있는 모든 명령이 원격 서버에서 실행됩니다.
- `ls -lhR local_dir/`: 첫 번째 명령입니다.
- `;`: 앞의 명령이 성공하든 실패하든 관계없이 다음 명령을 실행하도록 연결해 줍니다.
- `rclone lsl cloud_host:backup_dir/`: 두 번째 명령입니다.

### 논리 연산자 사용 (선택 사항)
만약 첫 번째 명령이 성공했을 때만 두 번째 명령을 실행하고 싶다면, 세미콜론 대신 논리 연산자 &&를 사용하세요.
```Bash
ssh -p host_port host_user@host_dir/ "ls -lhR local_dir/ && rclone lsl cloud_host:backup_dir/"
```
이 방식은 ls 명령이 오류 없이 정상적으로 완료되었을 때만 rclone 명령이 실행됩니다.

### 주의사항
- ssh 명령 뒤에 오는 원격 서버에서 실행할 명령어 전체는 반드시 **큰따옴표(")**로 묶어야 합니다. 이렇게 해야 ssh가 모든 명령을 하나의 인수로 인식하고 원격 서버로 전달할 수 있습니다.
- rclone 명령을 사용하려면, 원격 서버에 rclone이 미리 설치되어 있고, rclone 설정을 위한 .config/rclone/rclone.conf 파일이 올바르게 구성되어 있어야 합니다.
- 만약 원격 서버의 rclone 설정 파일 경로가 기본 경로와 다르거나 환경 변수에 설정되어 있지 않다면, 명령을 다음과 같이 수정해야 할 수도 있습니다.
```Bash
ssh -p host_port host_user@host_dir/ "ls -lhR local_dir/ ; rclone --config /path/to/your/rclone.conf lsl cloud_host:backup_dir/"
```
이 방법을 사용하면 별도의 스크립트 파일을 만들지 않고도 단일 bash 명령으로 원격 서버에서 두 가지 작업을 순차적으로 수행할 수 있습니다. 🚀

## 🔥 ubuntu 에서 virtualbox 7.1.12 를 설치하고
여기에 ubuntu 24.04.3 server 를 인스콜 하려고 하는데 `VirtualBox can't operate in VMX root mode. Please disable the KVM kernel extension, recompile your kernel and reboot (VERR_VMX_IN_VMX_ROOT_MODE).` 이런 에러가 나오네. 왜 그런거고, 어떻게 해결해야 하는거지?

🔋
### 🔋 VMX root mode 오류는 KVM(Kernel-based Virtual Machine)
이라는 리눅스 가상화 모듈과 VirtualBox가 동시에 **CPU 가상화 기능(VT-x/AMD-V)**을 사용하려고 충돌하면서 발생하는 문제입니다.

### 🛠️ 문제의 원인 및 해결 방법
VirtualBox는 KVM과 공존할 수 없습니다. 이 문제를 해결하기 위한 방법은 아래와 같습니다.

### 방법 1: KVM 모듈 비활성화 (권장)
가장 간단하고 권장되는 방법은 KVM 모듈을 일시적으로 비활성화하는 것입니다. 터미널에 다음 명령어를 입력하세요.
#### 1. KVM 모듈 언로드:
```Bash
sudo modprobe -r kvm_intel kvm
```
- `kvm_intel`: 인텔 CPU용 KVM 모듈
- `kvm`: AMD CPU용 KVM 모듈
- `modprobe -r`: 현재 로드된 커널 모듈을 제거합니다.
#### 2. VirtualBox 다시 실행:
이제 VirtualBox를 다시 시작하고 가상 머신을 실행해보세요.
### 방법 2: 부팅 시 KVM 모듈 자동 로드 방지
위 방법은 일시적인 해결책이므로, 시스템 재부팅 후에는 다시 KVM 모듈이 로드될 수 있습니다. 이를 방지하려면 다음 설정을 추가하세요.
#### 1. modprobe 설정 파일 생성:
```Bash
echo "blacklist kvm_intel" | sudo tee /etc/modprobe.d/blacklist-kvm.conf
echo "blacklist kvm" | sudo tee -a /etc/modprobe.d/blacklist-kvm.conf
```
- 이 명령은 /etc/modprobe.d/ 디렉토리에 blacklist-kvm.conf 파일을 생성하고, KVM 모듈이 부팅 시 자동으로 로드되는 것을 막습니다.
#### 2. 시스템 재부팅:
변경 사항을 적용하기 위해 시스템을 재부팅하세요.

### 💡 추가 정보
- `VirtualBox 6.1 버전 이전`: 일부 구 버전에서는 이 문제가 덜 발생했지만, 최근 VirtualBox는 KVM과의 충돌이 더 명확하게 나타납니다.
- `VirtualBox 7.0 버전 이상`: 최근 VirtualBox 버전에서는 KVM을 백엔드로 사용할 수 있는 기능(virtio)이 추가되었지만, 여전히 기본적으로는 KVM과의 충돌을 피하기 위해 KVM 모듈을 비활성화하는 것이 좋습니다.
### ⚠️ 주의 사항
- 이 방법은 Libvirt, KVM/QEMU를 사용하는 다른 가상화 솔루션(예: virt-manager)을 함께 사용하고 있지 않을 때만 적용하세요.
- 만약 KVM을 계속 사용해야 한다면, VirtualBox 대신 KVM/QEMU 기반의 솔루션(예: virt-manager)을 사용하는 것을 고려해 볼 수 있습니다. 이 경우 KVM과의 충돌이 전혀 발생하지 않습니다.

### 🔥 그러면, sudo modprobe -r kvm_intel kvm 명령으로 KVM 모듈을 언로드하고,
VirtualBox 를 다시 실행하면 된단 말이지? 그리고, VirtualBox 작업이 끝나면, 그냥 끝나면 되고? 그렇다면 좀 불편해도 이렇게 쓰면 되겠네. 그런데 KVM 모듈은 어떨때 쓰는건가? 난 ubuntu 에서 docker 와 mysql, gimp, java, python 정도만 쓰고 있어서 KVM 모듈과는 관련이 없을것 같은데 말이야.

🔋
### 🔋 네, 맞습니다. sudo modprobe -r kvm_intel kvm 명령어를 사용하면
KVM 모듈이 현재 세션에서만 언로드(unload)되므로, VirtualBox를 사용한 작업이 끝나면 컴퓨터를 재부팅하거나 해당 모듈을 다시 로드할 필요 없이 그대로 두어도 괜찮습니다. 편리하게 사용하시면 됩니다.

#### 🖥️ KVM 모듈은 왜 필요한가요?
**KVM(Kernel-based Virtual Machine)**은 리눅스 커널에 내장된 가상화 기술입니다. CPU의 가상화 확장 기능(Intel VT-x 또는 AMD-V)을 활용하여 리눅스에서 가상 머신(Virtual Machine)을 만들고 구동하는 데 사용됩니다.
- VirtualBox와는 다르게, KVM은 운영체제(커널) 자체에 통합되어 있어 하드웨어 가상화 성능을 최대한 활용할 수 있습니다.
- 주로 QEMU, virt-manager와 같은 가상화 관리 도구와 함께 사용됩니다.
#### KVM이 자동으로 설치되는 이유
우분투와 같은 리눅스 배포판은 KVM 모듈을 기본적으로 포함하거나, 가상화를 지원하는 다른 패키지들의 의존성 때문에 설치되는 경우가 많습니다.
- 예를 들어, libvirt-daemon-system 같은 패키지를 설치할 때 KVM 관련 패키지가 함께 설치되면서 해당 모듈이 자동으로 로드될 수 있습니다.
- 사용자는 직접 가상 머신을 만들지 않아도, 다른 시스템 도구들이 내부적으로 가상화 기능을 활용하기 위해 KVM 모듈이 필요할 수 있습니다.
#### 사용자 환경에서의 KVM
질문자님께서 사용하시는 Docker, MySQL, GIMP, Java, Python 등은 KVM과 직접적인 관련이 없습니다.
- Docker는 컨테이너 기술로, 가상 머신과 달리 리눅스 커널을 공유하며 격리된 환경을 만듭니다. KVM과는 기술적으로 다릅니다.
- 다른 애플리케이션들은 대부분 가상화 기술이 필요 없는 일반적인 프로그램들입니다.

따라서, 질문자님의 현재 사용 환경에서는 KVM 모듈이 필요하지 않으므로, VirtualBox를 사용할 때마다 일시적으로 언로드하는 방식이 가장 적절하고 안전한 해결책입니다.

## 🔥 특정 문자열있는 파일들을 vi 로 수정하기
지정한 디렉토리 "~/bin" 에서 특정 문자열 "lll=" 이 있는 파일들만 골라내서 그 각각의 파일을 vi 로 수정하는 ubuntu 에서의 bash 파일을 만들어줘.

🔋
### 🔋 주어진 요청에 따라 ~/bin 디렉토리에서 특정 문자열 "lll="을 포함하는 파일들만 찾아서
vi 에디터로 열어 수정할 수 있도록 하는 bash 스크립트를 작성해 드릴게요.

아래 스크립트는 두 가지 주요 기능을 수행합니다.
1. ~/bin 디렉토리 내의 모든 파일을 재귀적으로 탐색합니다.
2. 각 파일에서 "lll=" 문자열이 포함되어 있는지 확인합니다.
3. 문자열이 포함된 파일들을 찾아서 vi 에디터로 순차적으로 열어줍니다. 하나의 파일을 저장하거나 종료( :wq 또는 :q )하면 다음 파일을 자동으로 열어줍니다.

#### 스크립트 파일
```Bash
#!/bin/bash

SEARCH_DIR="$HOME/bin"
echo "#-- 검색할 디렉토리 경로 [ ${SEARCH_DIR} ]"
SEARCH_STRING="lll="
echo "#-- 찾을 문자열 [ ${SEARCH_STRING} ]"
# find 명령어를 사용하여 지정된 디렉토리 내에서 특정 문자열을 포함하는 파일들을 찾습니다.
# -type f : 파일만 검색합니다.
# -exec grep -q "$SEARCH_STRING" {} \; : 각 파일을 대상으로 grep을 실행하여 문자열이 있는지 확인합니다.
# -print0 : 파일 경로를 null 문자로 구분하여 출력합니다.
# xargs -0 : null 문자로 구분된 입력을 받아 vi 명령어의 인자로 전달합니다.
find "$SEARCH_DIR" -type f -exec grep -q "$SEARCH_STRING" {} \; -print0 | xargs -0 vi
```

#### 사용법
1. 위 코드를 복사하여 edit_files.sh 와 같은 이름으로 파일을 생성합니다.
```Bash
vi edit_files.sh
```
2. 복사한 코드를 파일에 붙여넣고 저장합니다.
3. 스크립트 파일에 실행 권한을 부여합니다.
```Bash
chmod +x edit_files.sh
```
4. 스크립트를 실행합니다.
```Bash
./edit_files.sh
```
이 스크립트는 vi를 종료( :q )하거나 저장( :wq )할 때마다 자동으로 다음 파일로 넘어갑니다. 모든 파일을 수정한 후에는 스크립트가 종료됩니다.

### 🔋 안녕하세요! Bash 스크립트에서 배열을 사용하다가 `Syntax error: "(" unexpected` 오류가 발생하셨군요.
이 오류는 일반적으로 스크립트가 Bash가 아닌 sh 쉘로 실행될 때 발생합니다.

`A_DOCKER_ROOT=("${qq}-1: 도커의 루트 디렉토리:" "${DockerRootDir}")` 이 구문은 Bash에서 배열을 선언하는 올바른 문법입니다. 하지만 sh는 Bash보다 훨씬 기본적인 쉘이기 때문에 이 문법을 이해하지 못합니다. 그래서 괄호 (를 예상치 못한 문자로 인식하여 오류를 내뱉는 것입니다.

#### 해결 방법
이 문제를 해결하는 방법은 크게 두 가지가 있습니다.

#### 1. 스크립트 실행 시 bash로 명시
스크립트를 실행할 때 sh restore.sh ... 대신 bash restore.sh ...로 실행하면 됩니다.
```Bash
bash restore.sh wiki.sql.7z
```
이렇게 하면 스크립트가 sh가 아닌 bash 쉘에서 실행되어 배열 문법을 올바르게 해석할 수 있습니다.

#### 2. 스크립트 상단에 shebang(셔뱅) 추가
스크립트 파일(restore.sh)의 맨 첫 줄에 shebang을 추가하여 이 스크립트가 bash로 실행되어야 함을 명시하는 방법입니다.
```Bash
#!/bin/bash

# A_DOCKER_ROOT=("${qq}-1: 도커의 루트 디렉토리:" "${DockerRootDir}")

```

#### 추가 정보: sh와 bash의 차이
- sh: Bourne Shell의 약자로, 유닉스 계열 시스템에서 가장 기본적인 쉘입니다. 호환성이 매우 높지만, 기능이 제한적입니다.
- bash: Bourne-Again Shell의 약자로, sh의 기능을 확장하고 더 많은 기능(배열, 연관 배열, 더 복잡한 제어 구조 등)을 추가한 현대적인 쉘입니다. 대부분의 리눅스 시스템에서 기본 쉘로 사용됩니다.

질문하신 배열 문법은 bash에서만 지원하는 기능이므로, 스크립트를 bash 환경에서 실행해야 오류가 발생하지 않습니다.

### 🔋 해당 문제는 Orange Pi에서 사용하는 Ubuntu의 기본 쉘 환경 설정과
관련이 있을 가능성이 높습니다.

#!/bin/bash를 추가했음에도 불구하고 sh로 실행했을 때와 동일한 오류가 나는 이유는, 스크립트 실행 시 sh가 가리키는 대상이 여전히 bash가 아닌 다른 쉘(예: dash)일 수 있기 때문입니다.

대부분의 우분투 시스템에서는 sh가 /bin/sh를 가리키고, 이 /bin/sh는 /bin/bash의 심볼릭 링크입니다. 그러나 일부 임베디드 시스템이나 최적화된 시스템에서는 성능상의 이유로 sh가 dash (Debian Almquist Shell)의 심볼릭 링크로 설정되어 있는 경우가 많습니다. dash는 bash보다 가볍고 빠르지만, bash에서만 지원하는 문법 (배열 선언 등)을 지원하지 않습니다.

bash restore.sh로 실행하면 sh가 아닌 bash 바이너리를 직접 호출하게 되므로, 배열 문법이 올바르게 해석되어 에러 없이 실행되는 것입니다.

#### 해결 방법
#### 1. sh가 어떤 쉘을 가리키는지 확인:
터미널에서 다음 명령어를 실행하여 /bin/sh가 어떤 쉘의 심볼릭 링크인지 확인해 보세요.
```Bash
ls -l /bin/sh
```
만약 결과가 ... /bin/sh -> dash와 같이 나온다면, sh는 dash를 사용하고 있다는 의미입니다.
#### 2. 심볼릭 링크 변경 (권장하지 않음):
sh를 bash로 변경하는 방법도 있지만, 시스템 안정성에 영향을 줄 수 있으므로 권장하지 않습니다. 많은 시스템 스크립트들이 dash의 경량성을 기반으로 작성되었기 때문에 sh를 bash로 바꾸면 예상치 못한 문제가 발생할 수 있습니다.
#### 3. 스크립트 실행 방식 통일:
가장 안전하고 확실한 방법은 모든 스크립트를 항상 **bash 스크립트명**으로 명시하여 실행하는 것입니다.
```Bash
bash restore.sh wiki.sql.7z
```
결론적으로, OS 자체의 문제라기보다는 Orange Pi Ubuntu의 기본 쉘 환경 설정이 데스크톱 Ubuntu와 다르기 때문에 발생하는 현상입니다. uname -a로 나오는 커널 버전 자체에는 문제가 없으며, sh가 bash가 아닌 dash를 사용하고 있기 때문에 #!/bin/bash가 제대로 동작하지 않는 것입니다. 스크립트를 실행할 때 항상 bash를 명시하는 것을 습관화하는 것이 좋습니다.

### 🔋 Docker Compose는 여러 컨테이너로 구성된 애플리케이션을 정의하고
실행하는 도구입니다. 🐳

### 컨테이너 오케스트레이션 도구
Ubuntu가 컴퓨터의 운영 체제(OS)인 것처럼, Docker Compose는 Docker 컨테이너를 관리하는 도구입니다. 좀 더 구체적으로, Docker Compose는 여러 개의 컨테이너를 하나로 묶어 관리하는 컨테이너 오케스트레이션(Container Orchestration) 도구의 일종이라고 할 수 있습니다.

예를 들어, 웹 애플리케이션을 배포할 때 웹 서버(Nginx), 데이터베이스(MySQL), 백엔드 서버(Node.js) 이렇게 3개의 컨테이너가 필요하다면, Docker Compose는 이 세 컨테이너의 설정(이미지, 네트워크, 볼륨 등)을 **docker-compose.yml**이라는 단일 파일에 정의하고, 이 파일을 이용해 한 번에 이 세 컨테이너를 실행하거나 중지할 수 있도록 해줍니다.

따라서 "Docker Compose는 컨테이너 여러 개를 한 번에 쉽게 관리하는 도구"라고 설명하면 이해하기 쉽습니다.

**Wiki.js는 위키 엔진(Wiki Engine) 또는 위키 소프트웨어(Wiki Software)**라고 부르는 게 가장 정확합니다.

간단히 말해, **웹 기반 위키를 만들고 관리하는 애플리케이션(Application) 또는 프로그램(Program)**이라고 생각하면 됩니다.

- 패키지(Package): 이 용어도 사용할 수는 있습니다. 특히 프로그래밍 환경에서 특정 기능을 수행하는 코드 묶음을 의미할 때 사용합니다. 하지만 Wiki.js의 전체적인 정체성을 나타내기보다는, "설치 가능한 소프트웨어 묶음"이라는 측면에 초점을 맞춘 표현이라서, Wiki.js의 본질적인 용도는 잘 드러나지 않습니다.
- 플랫폼(Platform): 이 용어는 여러 애플리케이션을 구동하거나 개발할 수 있는 기반 시스템을 의미합니다. Wiki.js는 그 자체로 플랫폼이라기보다는, 다른 사람들이 콘텐츠를 만들고 협업할 수 있는 플랫폼의 역할을 하는 애플리케이션이라고 보는 것이 더 정확합니다.

따라서 Wiki.js의 핵심 기능과 용도를 잘 설명하려면 **"위키 엔진"**이나 **"위키 소프트웨어"**라고 부르는 것이 가장 적절합니다. 📖

#### 2단계: 백업 파일 복원하기
이제 백업 파일을 PostgreSQL 컨테이너로 복원하는 스크립트입니다.
```Bash
#!/bin/bash

# PostgreSQL 컨테이너 이름 (docker-compose.yml에 정의된 이름)
DB_CONTAINER_NAME="wikijs_db"

# 백업 파일 경로
BACKUP_DIR="$HOME/backup"
BACKUP_FILE=$(ls -t "${BACKUP_DIR}"/wikijs-*.7z | head -n 1)

if [ -z "$BACKUP_FILE" ]; then
    echo "백업 파일($BACKUP_DIR/wikijs-*.7z)을 찾을 수 없습니다. 경로를 확인하세요."
    exit 1
fi

echo "백업 파일: $BACKUP_FILE 을 복원합니다."

# 7z 압축 해제
7za e -so "$BACKUP_FILE" > "/tmp/wikijs_restore.sql"

# PostgreSQL 컨테이너에 SQL 파일 복원
docker exec -i "$DB_CONTAINER_NAME" psql -U wikijs -d wikijs < "/tmp/wikijs_restore.sql"

echo "데이터베이스 복원이 완료되었습니다."

# 임시 파일 삭제
rm "/tmp/wikijs_restore.sql"
```

### 🔋 docker exec 명령에서 -U와 -d에 있는 'wikijs'는 각각 다른 것을 의미해요.

#### psql 명령 옵션 설명
psql은 PostgreSQL 데이터베이스에 접속하는 명령어입니다. 이 명령 뒤에 붙는 옵션들은 데이터베이스 접속 정보를 지정하는 역할을 합니다.
- psql -U wikijs: 여기서 -U는 User의 약자입니다. 뒤에 오는 wikijs는 데이터베이스에 접속할 때 사용할 사용자 이름을 의미합니다.
- psql -d wikijs: 여기서 -d는 Database의 약자입니다. 뒤에 오는 wikijs는 접속하려는 데이터베이스의 이름을 의미합니다.

즉, docker exec -i "$DB_CONTAINER_NAME" psql -U wikijs -d wikijs라는 명령어는 "$DB_CONTAINER_NAME" 컨테이너 안에서 wikijs라는 사용자로 wikijs라는 데이터베이스에 접속해서 백업 파일을 복원하라는 뜻입니다.

이러한 사용자 이름과 데이터베이스 이름은 docker-compose.yml 파일의 environment 변수에 이미 설정해 놓았기 때문에 똑같이 사용하면 됩니다. 📖

### 🔋 네, environment 변수는 컨테이너 내부에서 실행되는 애플리케이션의 설정값을 정할 때 사용합니다.
스크립트를 만들 때도 이 변수들을 활용할 수 있습니다.

#### environment 변수의 역할
docker-compose.yml 파일에서 정의한 environment 변수는 다음과 같은 용도로 사용됩니다.
- 컨테이너 내부 설정: 컨테이너가 시작될 때, 그 안에 있는 프로그램은 이 변수들을 읽어서 자체 설정을 적용합니다. 예를 들어, POSTGRES_USER와 POSTGRES_PASSWORD 변수는 PostgreSQL 컨테이너가 데이터베이스 사용자 계정을 만들 때 사용합니다.
- 애플리케이션 연결 정보: wiki.js 컨테이너는 DB_USER나 DB_PASS 같은 변수를 통해 PostgreSQL 컨테이너에 접속하는 데 필요한 정보를 얻습니다.

#### 스크립트에서 활용하기

### 🔥 docker-compose 를 쓸때 앞에 sudo 를 붙이지 않아도 되나?
나는 습관처럼 sudo 를 붙여왔는데말야.

### 🔋 네, docker-compose를 실행할 때 sudo를 붙이지 않아도 됩니다.

sudo 없이 Docker 명령어를 실행하려면, 현재 사용자 계정을 docker 그룹에 추가하면 됩니다. 이 작업을 완료하면 더 이상 sudo를 붙이지 않아도 되어 편리합니다.

아래는 현재 사용자를 docker 그룹에 추가하는 방법과 적용하는 절차입니다.

### 1. 현재 사용자를 docker 그룹에 추가
터미널에서 아래 명령어를 실행하세요.
```Bash
sudo usermod -aG docker $USER
```
이 명령어는 현재 로그인한 사용자($USER)를 docker 그룹에 추가합니다. -aG 옵션은 사용자를 그룹에 추가(append)하고, 그룹 멤버십을 유지(group)하겠다는 의미입니다.

### 2. 변경 사항 적용
명령어를 실행한 후에는 로그아웃했다가 다시 로그인하거나, 아래 명령어를 실행하여 그룹 변경 사항을 즉시 적용해야 합니다.
```Bash
newgrp docker
```
newgrp docker 명령어는 현재 쉘의 그룹 멤버십을 docker 그룹으로 변경합니다.

### 3. 확인
위 절차를 완료한 후에는 sudo 없이 docker run hello-world와 같은 간단한 명령어를 실행하여 정상적으로 작동하는지 확인해 보세요.
```Bash
docker run hello-world
```
만약 정상적으로 실행된다면, docker-compose 명령어 역시 sudo 없이 사용할 수 있습니다.

### 🔥 docker-compose down 명령은 db 가 멈추는거지
db 내용을 삭제하는것은 아니지 않나? 만일 db 를 삭제하는 명령이라면, 이름을 바꾸고나서 데이터를 다시 리스토어 해야 할것 아닌가?

### 🔋 docker-compose down 명령어는 컨테이너를 중지하고 삭제하지만,
기본적으로 볼륨(volume)에 저장된 데이터는 삭제하지 않습니다. 따라서 DB 데이터는 유지됩니다.

#### docker-compose down의 동작
docker-compose down 명령은 docker-compose.yml 파일에 정의된 모든 서비스의 컨테이너를 중지하고 삭제하는 역할을 합니다. 하지만 이 명령은 기본적으로 **익명 볼륨(anonymous volumes)**만 삭제하며, **명시적으로 이름이 지정된 볼륨(named volumes)**이나 **호스트 디렉토리에 마운트된 볼륨(bind mounts)**은 삭제하지 않습니다.

제공하신 docker-compose.yml 파일에서 DB 컨테이너의 volumes 설정은 다음과 같습니다.
```YAML
volumes:
  - /home/docker/pgsql:/var/lib/postgresql/data
```
이것은 호스트 머신의 /home/docker/pgsql 디렉토리를 컨테이너의 /var/lib/postgresql/data 디렉토리에 마운트하는 바인드 마운트(bind mount) 방식입니다. 즉, PostgreSQL 데이터는 컨테이너 내부가 아닌 호스트 머신의 특정 디렉토리에 저장됩니다.

따라서 docker-compose down 명령어를 실행하더라도 /home/docker/pgsql 디렉토리에 있는 DB 데이터는 그대로 유지됩니다.

#### 이름 변경 후 데이터 유지 방법
컨테이너 이름, DB 이름, 유저, 비밀번호 등을 변경하는 경우, 데이터는 이미 /home/docker/pgsql에 존재하므로 별도의 백업 및 복원 과정이 필요 없습니다. 새로운 docker-compose.yml 파일로 컨테이너를 다시 시작하면, 새롭게 생성된 PostgreSQL 컨테이너가 기존 데이터가 있는 /home/docker/pgsql 디렉토리를 사용하게 됩니다.

그러나 DB 이름, 유저, 비밀번호가 변경되었으므로, 새로운 컨테이너가 시작될 때 기존 데이터베이스 설정과 충돌하지 않도록 몇 가지 추가적인 조치가 필요할 수 있습니다.

#### 해결책:

#### 1. 기존 DB 컨테이너 삭제 후 데이터베이스 설정 변경:
- 가장 안전한 방법은 docker-compose down으로 기존 컨테이너를 삭제하고, 새로운 docker-compose.yml 파일로 컨테이너를 다시 실행하는 것입니다.
- 이때 DB 유저명, DB명, 비밀번호를 변경했으므로, 새로운 설정으로 컨테이너를 실행하면 PostgreSQL이 기존 데이터와 새로운 설정 정보를 맞추려 합니다.
- PostgreSQL은 기존 데이터 디렉토리(/var/lib/postgresql/data)에 이미 데이터가 있을 경우, POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD 환경 변수 값을 무시하고 기존 설정을 유지하려는 경향이 있습니다. 따라서 새롭게 설정한 유저와 비밀번호로 접속이 되지 않을 수 있습니다.

#### 2. 데이터베이스 설정 변경 스크립트 실행 (권장):
- 가장 확실한 방법은 컨테이너를 재시작하기 전에, 새로운 설정(유저, 비밀번호 등)으로 기존 데이터를 업데이트하는 스크립트를 직접 실행하는 것입니다.
- 이 작업을 위해 pgAdmin과 같은 GUI 툴을 사용하거나, psql을 이용하여 PostgreSQL에 접속한 뒤 아래와 같은 SQL 쿼리를 실행할 수 있습니다.
```SQL
ALTER USER wikijs WITH PASSWORD 'wikipswd';
ALTER DATABASE wiki RENAME TO wikidb;
ALTER USER wikijs RENAME TO wikiuser;
```
이후 docker-compose.yml 파일의 DB_USER와 DB_NAME 값을 새로운 값으로 업데이트하면 됩니다.

결론적으로, docker-compose down은 DB 데이터를 삭제하지 않으므로 백업 및 복원 작업이 필수는 아닙니다. 하지만 DB 유저, 비밀번호, 이름을 변경할 때는 컨테이너가 재시작될 때 기존 데이터베이스의 설정을 새로운 설정과 일치시키기 위한 추가적인 작업이 필요할 수 있습니다. 가장 간단하고 안전한 방법은 백업 후 컨테이너를 재생성하고, 만약 데이터 불일치가 발생한다면 데이터베이스 설정을 직접 수정하는 것입니다.

### 🔋 docker exec를 사용하여 DB 설정을 변경하는 SQL 쿼리 스크립트를
만들어 드릴게요.

이 스크립트는 docker-compose down 명령어를 실행하기 전에 먼저 실행하여 기존 데이터베이스의 이름, 유저, 비밀번호를 변경하는 작업입니다.

### DB 설정 변경 bash 스크립트
아래 스크립트는 PostgreSQL 컨테이너가 실행 중인 상태에서 작동합니다. docker exec 명령어를 사용하여 컨테이너 내부로 들어가 psql을 통해 SQL 쿼리를 실행하는 방식입니다.

dbcon은 스크립트 실행 후 변경될 컨테이너 이름이므로, 스크립트 실행 전에 현재 사용 중인 DB 컨테이너 이름을 확인해야 합니다. 만약 현재 컨테이너 이름이 wikijsdb라면 wikijsdb로 변경하여 사용하세요.
```Bash
#!/bin/bash

# 현재 DB 컨테이너 이름 확인
CURRENT_DB_CONTAINER="wikijsdb" # 현재 사용 중인 DB 컨테이너 이름으로 변경하세요.

# 새로운 DB 설정 변수
NEW_DB_NAME="wikidb"
NEW_DB_USER="wikiuser"
NEW_DB_PASSWORD="wikipswd"

# 기존 DB 설정 변수
OLD_DB_NAME="wiki"
OLD_DB_USER="wikijs"

echo "기존 DB 설정을 업데이트합니다..."

seqno=3
# 1. DB 유저 이름 변경
echo "(${seqno}-1) DB 유저 '${OLD_DB_USER}'를 '${NEW_DB_USER}'로 변경합니다."
docker exec -it $CURRENT_DB_CONTAINER psql -U $OLD_DB_USER -d $OLD_DB_NAME -c "ALTER USER \"${OLD_DB_USER}\" RENAME TO \"${NEW_DB_USER}\";"

# 2. DB 유저 비밀번호 변경
echo "(${seqno}-2) DB 유저 '${NEW_DB_USER}'의 비밀번호를 변경합니다."
docker exec -it $CURRENT_DB_CONTAINER psql -U $NEW_DB_USER -d $OLD_DB_NAME -c "ALTER USER \"${NEW_DB_USER}\" WITH PASSWORD '${NEW_DB_PASSWORD}';"

# 3. 데이터베이스 이름 변경
echo "(${seqno}-3) 데이터베이스 '${OLD_DB_NAME}'를 '${NEW_DB_NAME}'으로 변경합니다."
docker exec -it $CURRENT_DB_CONTAINER psql -U $NEW_DB_USER -d postgres -c "ALTER DATABASE \"${OLD_DB_NAME}\" RENAME TO \"${NEW_DB_NAME}\";"

echo "((${seqno})) DB 설정 변경이 완료되었습니다."
```
